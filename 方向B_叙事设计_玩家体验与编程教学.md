# 方向 B ·《编程小镇：规则醒来的那一天》叙事设计、玩家体验与编程教学说明

> 目的：解决“现在的策划案看不懂玩法”的问题。  
> 这份文档不再从系统表开始，而是从**玩家如何进入故事、如何操作、如何理解编程、如何感到故事发生**来说明方向 B。

---

## 0. 一句话说明玩法

> 玩家不是学习写代码语法，而是在一个动物小镇里，用“如果……就……”的行为规则改变居民行动，然后观察居民如何误解、执行、协商和反抗这些规则。

最简单的例子：

```text
原规则：
如果有人饿了 → 阿橘烤面包

玩家改成：
如果有人饿了 → 阿橘唱歌安慰

结果：
阿橘真的去唱歌。
有居民被安慰。
但大家还是饿。
铃兰投诉：“歌不能登记为早餐。”
毛栗把这件事写成快讯。
洛洛开始做一个“会吐司的留声机”。
```

这就是本作的核心体验：

> **玩家改的是规则，故事从规则的后果里长出来。**

---

## 1. 这个游戏讲什么故事？

### 1.1 表层故事

鹳谷小镇是一个童话般的动物小镇。  
这里每个居民都有一套“生活规则”：

- 面包师阿橘：看到有人饿了，就烤面包。
- 邮差毛栗：收到消息，就立刻传播。
- 镇长助理铃兰：发现混乱，就制定表格。
- 修理工洛洛：机器坏了，就慢慢修。

某天，小镇的旧规则系统开始松动。  
鹳先生把玩家请来，担任新的“规则管理员”。

玩家每天能修改少量居民规则，帮助小镇处理问题。

### 1.2 深层故事

这个故事真正想讲的是：

> 规则可以安排行为，但不能完全定义一个人。

一开始，玩家会以为居民只是“执行规则的角色”。

但随着玩家不断修改规则，居民会开始：

- 记得自己被改过；
- 对规则产生疑惑；
- 因为别人的反应改变心情；
- 逐渐表达“我不只是这张卡”的自我意识。

所以故事不是“玩家把小镇调好”。  
故事是：

> 玩家在调试小镇时，发现规则背后住着真正的居民。

---

## 2. 玩家扮演谁？

玩家扮演：

> **鹳谷小镇的新任规则管理员。**

但这个身份不是程序员、老师或神。

玩家有三个限制：

1. **不能直接命令居民**  
   不能点“阿橘去烤面包”。

2. **不能直接改结局**  
   不能选择“小镇幸福”。

3. **只能修改行为规则**  
   例如把“看到饥饿 → 烤面包”改成“看到饥饿 → 唱歌安慰”。

因此玩家的幻想不是“控制一切”，而是：

> 我像在调试一个活的系统。  
> 我能改规则，但必须承担规则被活人理解后的后果。

## 3. 玩家第一局具体怎么玩？

第一局建议叫：

# 第一日：早餐危机

### 3.1 开场 0-2 分钟：小镇出问题

玩家看到一张工单：

```text
今日工单：早餐危机

面包炉坏了。
镇民很快会饿。
你不能直接发面包，也不能直接修炉。
请通过修改居民规则，让小镇撑过今天。
```

画面上有 4 个居民：

| 居民 | 当前状态 |
|---|---|
| 阿橘 | 抱着失败面团，很慌 |
| 毛栗 | 想把“面包炉坏了”写成快讯 |
| 铃兰 | 正在准备早餐排队表 |
| 洛洛 | 说“我可以修，但不要催” |

玩家此时理解：

> 小镇不是坏在一个敌人身上，而是坏在一组规则和关系里。

---

### 3.2 教学 2-5 分钟：认识 if-then 规则

系统展示阿橘的默认行为卡：

```text
如果看到有人饥饿 → 烤面包
```

鹳先生解释：

```text
这不是一句台词。
这是阿橘平时生活的规则。
你可以改它，但阿橘会用自己的方式理解它。
```

玩家可选几张替换卡：

| 原行动 | 可替换行动 |
|---|---|
| 烤面包 | 唱歌安慰 |
| 烤面包 | 分发失败面团 |
| 烤面包 | 求洛洛帮忙 |

玩家第一次不需要懂复杂系统，只要知道：

```text
如果 A 发生 → 角色做 B
```

这就是最基础的编程概念：条件与行为。

---

### 3.3 第一次操作 5-8 分钟：改一条规则

玩家可能选择：

```text
如果看到有人饥饿 → 阿橘唱歌安慰
```

再加一个语气标签：

```text
隆重地
```

最终规则变成：

```text
如果看到有人饥饿 → 阿橘隆重地唱歌安慰
```

玩家点击：

```text
运行上午
```

---

### 3.4 上午 8-14 分钟：规则变成故事

系统开始结算。

玩家看到的不是代码输出，而是一段小镇事件：

```text
阿橘站上面粉箱，清了清嗓子。

阿橘：
“如果面包还没醒，那我先让大家的胃听一首前奏！”

饱腹 -6
亲密 +8
荒诞 +6
```

接着其他 Agent 反应：

```text
铃兰：
“阿橘，歌不能登记为早餐。”

毛栗：
“我已经把《早餐歌危机》写进今日快讯了！”

洛洛：
“如果歌不能当早餐，也许我可以做一个会吐司的留声机。”
```

玩家立刻理解：

> 我的规则生效了。  
> 但它带来的后果不是单一的。  
> 它解决了一些问题，也制造了另一些问题。

这就是玩法。

---

### 3.5 下午 14-20 分钟：玩家开始调试

玩家看到小镇状态：

| 指标 | 变化 |
|---|---|
| 饱腹 | 45 → 38 |
| 秩序 | 65 → 61 |
| 亲密 | 50 → 64 |
| 荒诞 | 20 → 47 |

玩家明白：

- 大家更亲近了；
- 但还是饿；
- 小镇变得更荒诞。

此时玩家可以再改一张规则。

例如改洛洛：

```text
原规则：
如果机器坏了 → 慢慢修

新规则：
如果机器坏了 → 做替代工具
```

再运行下午。

结果可能出现：

```text
洛洛做出了“会吐司的留声机”。
它一边播放阿橘的早餐歌，一边吐出半熟面包片。

饱腹 +9
荒诞 +10
阿橘对洛洛关系 +8
铃兰秩序 -4
```

玩家体验到：

> 第二条规则不是孤立的。  
> 它和第一条规则发生了组合效果。

这就是“编程里的组合逻辑”，但以故事呈现。

---

### 3.6 夜晚 20-30 分钟：故事回声

夜晚玩家不能再改规则，只能看小镇如何消化这一天。

夜报显示：

```text
今日结果：温暖但混乱的小镇

因果链：
你把阿橘的“烤面包”改成“唱歌安慰”
→ 阿橘在广场唱早餐歌
→ 毛栗传播“早餐歌危机”
→ 铃兰认为歌不能算早餐
→ 你让洛洛做替代工具
→ 洛洛做出会吐司的留声机
→ 大家吃到了半熟面包，也记住了那首歌
```

Agent 日记：

```text
阿橘：
今天我第一次觉得，也许我不只是会烤面包。

洛洛：
那个留声机不算修理委托，但它是我的发明。

铃兰：
小镇没有按表运行。可大家好像没有那么害怕。

毛栗：
明天的标题我已经想好了：《规则醒来的那一天》。
```

这一刻故事完成：

> 玩家不是看了一段剧情，而是读到了自己规则造成的后果。

## 4. 这个游戏和“编程教学”有什么关系？

### 4.1 它不教语法

本作不教：

- Python
- JavaScript
- Scratch 积木细节
- 变量声明
- 循环语法
- 函数写法

如果把它做成语法教学，会变得很硬，也会失去故事感。

### 4.2 它教“编程思维”

本作真正教学的是这些概念：

| 编程概念 | 游戏中的表现 |
|---|---|
| 条件判断 if | 如果看到有人饥饿 |
| 行为执行 then | 就烤面包 / 唱歌 / 修理 |
| 参数 | 温柔地 / 隆重地 / 秘密地 |
| 状态变量 | 饱腹、秩序、亲密、荒诞 |
| 因果链 | 你改了 X → 发生 Y → 影响 Z |
| 调试 | 第一次改坏了，下午再修正 |
| 副作用 | 唱歌提升亲密，但降低饱腹 |
| 组合逻辑 | 阿橘唱歌 + 洛洛替代工具 → 会吐司的留声机 |
| Agent / 对象 | 每个居民有自己的规则、记忆和状态 |
| 涌现 | 多条简单规则组合出意外故事 |

### 4.3 为什么用故事教编程？

因为初学者最难理解的不是语法，而是：

> 我写下的规则，为什么会产生我没想到的后果？

传统教学常说：

```text
if hungry:
    bake_bread()
```

玩家可能觉得这只是代码。

但在游戏里，它变成：

```text
如果有人饿了，阿橘就烤面包。
```

当玩家把它改成：

```text
如果有人饿了，阿橘就唱歌。
```

玩家会马上看到：

- 情绪被安慰了；
- 饱腹没有解决；
- 其他居民开始回应；
- 小镇系统出现副作用。

这比抽象讲“条件、行为、副作用、调试”更直观。

### 4.4 教学目标不是“会写代码”，而是“理解规则系统”

本作的教育定位应该是：

> 用叙事沙盒让玩家理解编程思维：规则、条件、状态、调试、组合和副作用。

它更像：

- 编程启蒙；
- 系统思维训练；
- AI Agent 行为实验；
- 规则叙事游戏。

而不是：

- 编程刷题；
- Scratch 模拟器；
- 代码课程。

## 5. Storytelling：故事如何被讲出来？

### 5.1 不是线性剧情，而是“规则叙事”

传统叙事：

```text
作者写好剧情 → 玩家阅读剧情
```

本作叙事：

```text
设计师设定小镇问题
→ 玩家改规则
→ Agent 解释并执行规则
→ 其他 Agent 回应
→ 系统记录后果
→ 夜报整理成故事
```

所以故事不是预先写死的长文本，而是由四层生成：

| 层级 | 负责什么 |
|---|---|
| 工单 | 给今天的戏剧冲突 |
| 行为卡 | 给玩家可修改的规则 |
| Agent | 让规则变成有性格的行动 |
| 夜报 | 把行动后果整理成可理解的故事 |

### 5.2 每局故事结构

每一局仍然有起承转合：

| 结构 | 玩家体验 | 早餐危机例子 |
|---|---|---|
| 起 | 小镇出问题 | 面包炉坏了 |
| 承 | 玩家改第一条规则 | 阿橘不烤面包，改唱歌 |
| 转 | 规则产生副作用 | 大家开心但仍然饿 |
| 合 | 玩家二次调试，夜报总结 | 洛洛做吐司机，形成温暖但混乱结局 |

这让沙盒不是散的，而是有故事节奏。

### 5.3 戏剧冲突来自哪里？

冲突来自三种张力：

#### 1. 规则 vs 需求

阿橘被改成唱歌，但大家真正需要早餐。

#### 2. 规则 vs 人设

铃兰可以被改成“放假”，但她天生焦虑，会担心小镇失序。

#### 3. 规则 vs 自我

洛洛本来只修机器，但当他做出“会吐司的留声机”，他第一次觉得这是自己的发明。

这三种张力让故事成立。

## 6. 玩家每一步在想什么？

### 6.1 玩家不是“选最优解”，而是在权衡代价

比如早餐危机里：

| 选择 | 好处 | 代价 |
|---|---|---|
| 让阿橘继续烤面包 | 饱腹稳定 | 没有惊喜，阿橘仍是工具人 |
| 让阿橘唱歌 | 亲密上升 | 饱腹下降 |
| 让毛栗传播消息 | 大家知道问题 | 谣言和荒诞上升 |
| 让铃兰开会 | 秩序上升 | 事情变慢 |
| 让洛洛快修 | 可能解决炉子 | 疲劳上升，可能失败 |
| 让洛洛做替代工具 | 产生新故事 | 荒诞上升 |

玩家的乐趣来自：

> 我知道没有完美规则，但我想看看这条规则会把小镇带到哪里。

### 6.2 玩家反馈来自三处

| 反馈 | 作用 |
|---|---|
| 小镇指标 | 告诉玩家系统层面发生了什么 |
| Agent 群聊 | 告诉玩家居民如何理解这件事 |
| 夜报因果链 | 告诉玩家自己造成了什么故事 |

如果只有指标，会像经营表格。  
如果只有群聊，会像聊天机器人。  
如果只有夜报，会像旁白总结。

三者合起来，玩家才看懂玩法。

## 7. 为什么需要多 Agent？

如果只有一个 NPC，玩法会变成：

```text
玩家改规则 → NPC 执行 → 看结果
```

这像一个玩具。

多 Agent 的价值在于：

```text
玩家改阿橘
→ 阿橘影响铃兰
→ 铃兰影响秩序
→ 毛栗传播这件事
→ 洛洛做出替代方案
→ 阿橘因此改变对自己的理解
```

也就是：

> 玩家改一条规则，但故事发生在关系网里。

这才是多 Agent 的核心展示。

---

## 8. 为什么玩家会产生情感？

情感不来自“动物很可爱”本身，而来自玩家看到：

1. Agent 有自己的需求；
2. 玩家规则影响了它；
3. 它记住了这件事；
4. 它和别人发生关系变化；
5. 它在夜晚写下自己的理解。

例如阿橘日记：

```text
我以前以为，大家喜欢我是因为我会烤面包。
今天他们饿着肚子听我唱完了歌。
这好像更糟，又好像不是。
```

这句话让玩家意识到：

> 我不是只改了一个 NPC 功能，我改变了它对自己的理解。

这就是情感核心。

## 9. 设计上要避免什么？

### 9.1 不要做成编程课

不要出现：

```text
今天我们学习 if 语句。
if 的中文意思是如果。
```

应该让玩家自然理解：

```text
如果有人饿了 → 阿橘会做什么？
```

### 9.2 不要做成自由聊天

如果玩家主要是在和 NPC 闲聊，编程概念就弱了。

核心必须始终是：

```text
改规则 → 跑时段 → 看后果
```

### 9.3 不要做成纯数值经营

如果玩家只关心饱腹和秩序，就会失去故事。

每次数值变化都要有 Agent 解释。

### 9.4 不要做成全随机涌现

玩家必须能从夜报里看懂：

```text
因为我改了什么，所以发生了什么。
```

涌现要有惊喜，但不能没有因果。

## 10. 首局完整体验脚本

下面是可以直接给开发或评委看的首局流程。

### 10.1 开场

```text
鹳先生：
欢迎来到鹳谷小镇。
这里每位居民都有自己的生活规则。
规则让小镇运转，也让小镇有点僵硬。

今天面包炉坏了。
你不能直接修炉，也不能直接发早餐。
你只能改一点点规则。
我们看看小镇会怎么醒来。
```

### 10.2 玩家看到默认规则

```text
阿橘：
如果看到有人饥饿 → 烤面包

毛栗：
如果听到新消息 → 写进快讯

铃兰：
如果发现混乱 → 制定表格

洛洛：
如果机器坏了 → 慢慢修
```

### 10.3 第一次改规则

玩家把阿橘改成：

```text
如果看到有人饥饿 → 隆重地唱歌安慰
```

### 10.4 上午结果

```text
阿橘站上面粉箱，唱起了早餐歌。

铃兰：
歌不能登记为早餐。

毛栗：
但可以登记为头条！

洛洛：
我在想，吐司机能不能也唱歌。
```

指标：

```text
饱腹下降
亲密上升
荒诞上升
```

### 10.5 第二次改规则

玩家把洛洛改成：

```text
如果机器坏了 → 做替代工具
```

### 10.6 下午结果

```text
洛洛没有修好面包炉。
但他做出一台会唱歌的吐司机。

吐司机吐出的面包半熟，但每片都有节奏。

阿橘：
这比我的歌更像早餐。

铃兰：
我需要新增一个分类：可食用音乐设备。
```

指标：

```text
饱腹回升
秩序下降
亲密上升
荒诞大幅上升
```

### 10.7 夜报

```text
今日结果：温暖但混乱的小镇

你没有让小镇恢复正常。
但你让大家用不正常的方式撑过了早餐。
```

Agent 日记：

```text
阿橘：
如果面包能唱歌，那我也许不只是面包师。

洛洛：
我今天没有修好炉子。
但我做了一个以前不存在的东西。
```

这就是第一局故事。

## 11. 用一句话解释“玩法”

如果有人问“这游戏到底怎么玩”，回答：

> 每局小镇会出一个问题。玩家不能直接命令 NPC，只能修改他们的“如果……就……”行为规则。修改后运行一个时段，AI 居民会按各自性格解释规则、互相反应，小镇指标和关系随之变化。夜晚系统把玩家改动造成的因果链整理成夜报，让玩家看到自己如何把代码规则变成一段小镇故事。

## 12. 用一句话解释“编程教学”

如果有人问“这和编程有什么关系”，回答：

> 它不教语法，而是用故事让玩家体验编程思维：条件判断、行为执行、状态变化、副作用、调试、组合逻辑和多 Agent 涌现。

---

## 13. 用一句话解释“叙事设计”

如果有人问“故事怎么讲”，回答：

> 故事不是预先播给玩家的，而是从“工单问题 → 玩家改规则 → Agent 执行并互相回应 → 夜报回放因果链”这个流程中生成；玩家读到的是自己规则造成的后果。

---

## 14. 最终设计判断

方向 B 不应该被包装成：

```text
AI 编程教学小游戏
```

而应该被包装成：

```text
一个用编程规则讲童话小镇故事的多 Agent 叙事沙盒
```

它的核心卖点不是“学会 if 语句”，而是：

> 玩家第一次直观感受到：规则不是死的。  
> 当规则被有记忆、有关系、有需求的角色执行时，规则会变成故事。
