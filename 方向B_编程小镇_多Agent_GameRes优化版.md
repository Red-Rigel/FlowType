# 方向 B ·《编程小镇：规则醒来的那一天》多 Agent 规则沙盒（GameRes 优化版）

> 版本：v0.4 GameRes-informed playable design  
> 基于文件：`uploads/__B_______Agent____9eb4.md`  
> 参考方法论：`output/gameres_game_design/` 下 GameRes「游戏设计」91 篇文章方法论  
> 目标：保留“代码即社会契约 / 多 Agent 涌现”的高概念，把它改成 1-2 周能做、首局能玩懂、比赛展示能看出 AI 价值的 Web MVP。

---

## 0. 优化结论先行

方向 B 值得继续，而且比方向 A 更适合 AI / 多 Agent 赛道。  
但 v0.3 仍有一个核心风险：

> 它已经像一个完整系统设定，但首局玩家“为什么改、怎么判断改得好、改完到底看什么”还不够硬。

本版的关键修改是：

| 原 v0.3 | v0.4 优化 |
|---|---|
| 玩家每天观察小镇 | 玩家每天收到一张“规则工单” |
| 行为卡是玩法核心 | 行为卡 + 需求关系 + 可观察行动共同构成核心 |
| 4 个 Agent 各自执行规则 | 4 个 Agent 形成“需求与被需求”的小社会 |
| selfhood 是长期钩子 | selfhood 改为首局可见的“轻微越界” |
| 鹳先生夜报总结 | 夜报变成“事件回放 + Agent 日记 + 因果链” |
| 全局变量较多 | P0 收束为 4 个：饱腹、秩序、亲密、荒诞 |

## 1. 新定位

### 一句话

> **你是鹳谷小镇的新任规则管理员。每个动物居民都有 if-then 行为卡，但他们会记忆、会误解、会互相影响。你每天只能改少量规则，然后观察这个小社会如何把代码活成故事。**

### 比赛介绍版

> **一个多 Agent 规则沙盒：玩家修改 NPC 的行为规则，AI 居民根据人设、记忆、关系和当前环境解释规则、互相回应，并在规则之外逐渐长出自我。**

### 玩家幻想

> 我不是在写剧情，也不是在聊天。  
> 我像在调试一个活的小镇：改一条规则，四个居民就会误会、配合、抱怨、模仿，最后把我的规则变成我没预料到的故事。

## 3. 设计支柱

### 支柱 1：规则是输入，不是答案

玩家改的是 if-then 行为卡，但卡片只规定“倾向”。  
Agent 会根据：

- 自己的人设
- 当前需求
- 近期记忆
- 与其他 Agent 的关系
- 小镇状态

解释这条规则。

同一张卡，在不同 Agent 身上必须产生不同语义。

### 支柱 2：小镇是需求网络

参考“社交=需求与被需求”，4 个 Agent 必须互相需要。

| Agent | 能提供 | 会需要 | 关系张力 |
|---|---|---|---|
| 阿橘 | 面包、安慰、表演 | 被认可 | 不想只当工具人 |
| 毛栗 | 信件、消息、传播 | 秘密和信任 | 好奇心会破坏边界 |
| 铃兰 | 日程、秩序、规则解释 | 安全感 | 守规矩但怕无用 |
| 洛洛 | 修理、设施、发明 | 时间和理解 | 慢会被误会成懒 |

> 如果一个 Agent 的行为不会影响其他 Agent，它就不是多 Agent 核心，只是 NPC 装饰。

### 支柱 3：玩家目标来自工单，不来自自由实验

纯沙盒首局容易迷失。  
所以每局给玩家一张“规则工单”：

```text
今天小镇出了一个问题。
你不能直接命令居民，只能改少量行为卡。
请在一天内观察结果，并让小镇以某种方式解决问题。
```

### 支柱 4：状态机管事实，LLM 管解释

| 层级 | 谁负责 | 说明 |
|---|---|---|
| 行动是否合法 | 状态机 | Agent 只能从 allowed_actions 中选 |
| 数值如何变化 | 状态机 | 饱腹、秩序、亲密、荒诞 |
| 关系如何变化 | 状态机 + 规则表 | 行动组合产生关系 delta |
| Agent 怎么说 | LLM | 根据人设、记忆、行为生成短台词 |
| Agent 怎么记 | LLM 起草，状态机压缩 | 写入短期/长期记忆 |

### 支柱 5：小而深，首局必须看见 AI 价值

P0 不追求大沙盒，只追求一局里出现：

1. 玩家改规则；
2. Agent 不同理解规则；
3. Agent 互相引用和影响；
4. 小镇指标变化；
5. 某个 Agent 记住这次修改；
6. 夜报能回放因果链。

## 5. 核心玩法循环

### 5.1 Moment-to-moment loop：60-120 秒

```text
读工单
→ 查看小镇状态和 Agent 需求
→ 改 1 张行为卡 / 加 1 个语气标签
→ 运行一个时段
→ 观看 Agent 可观察行动
→ Agent 群聊互相回应
→ 小镇指标与关系变化
→ 事件进入因果日志
```

### 5.2 一局 25-35 分钟

| 阶段 | 时间 | 玩家动作 | 系统反馈 | 情绪目标 |
|---|---:|---|---|---|
| 规则管理员上任 | 0-3 min | 读一张简单工单 | 鹳先生说明“不能命令，只能改规则” | 轻快、好奇 |
| 第一次改卡 | 3-8 min | 改 2 张卡 | Agent 读新规则并表达困惑/期待 | “规则真的生效” |
| 上午运行 | 8-14 min | 观看行动与群聊 | 指标变化 + 事件卡 | 惊喜、好笑 |
| 下午修正 | 14-20 min | 再改 1 张卡或恢复旧卡 | Agent 记得玩家又改了 | 策略、责任 |
| 夜晚观察 | 20-28 min | 不再干预 | 自我值/关系触发一件小事件 | “他们像活的” |
| 夜报结算 | 28-35 min | 看因果链和日记 | 结果类型 + 明日种子 | 回味、复盘 |

## 6. 首局推荐工单

### 工单 A：早餐危机（推荐首局）

> 面包炉坏了，镇民早晨会饿。你不能直接修炉，也不能直接发面包，只能改居民的行为规则，让小镇撑过今天。

#### 为什么适合作首局

| 优点 | 说明 |
|---|---|
| 目标清晰 | 饱腹不要太低 |
| 规则直观 | 阿橘烤面包、洛洛修炉、铃兰维持秩序、毛栗传播消息 |
| 冲突自然 | 唱歌能提升亲密但不能填饱肚子 |
| 多 Agent 明显 | 修炉、安抚、排队、传话都能互相影响 |

#### 默认状态

| 指标 | 初始 |
|---|---:|
| 饱腹 | 45 |
| 秩序 | 65 |
| 亲密 | 50 |
| 荒诞 | 20 |

#### 初始事件

```text
面包炉冒出一团蓝烟。
阿橘抱着没发起来的面团发呆。
铃兰已经开始写“早餐应急排队表”。
毛栗想把这件事写成快讯。
洛洛说：“我可以修，但不能催。”
```

### 工单 B：一封不能读的信（P1）

主题：信息传播与信任。  
核心 Agent：毛栗。  
风险：过早引入隐私伦理，首局理解成本略高。

### 工单 C：钟楼慢了十分钟（P1）

主题：秩序与生活节奏。  
核心 Agent：铃兰、洛洛。  
风险：指标反馈不如饱腹直观。

## 7. Agent 设计（优化版）

### 7.1 Agent 规格

每个 Agent 都必须具备：

| 字段 | 作用 |
|---|---|
| visible_role | 玩家一眼能懂的社会功能 |
| hidden_need | 触发自我值的内在需求 |
| provides | 能满足别人什么需求 |
| depends_on | 需要别人什么支持 |
| hard_rules | 绝不突破的人设边界 |
| behavior_cards | 默认行为倾向 |
| visible_action_set | 可观察动作集合 |
| memory_slots | 记忆容量 |

### 7.2 四个 Agent

#### 阿橘 · 面包师

| 项 | 内容 |
|---|---|
| 可见功能 | 提供食物和情绪安慰 |
| 隐性需求 | 想被当作艺术家，而不只是厨子 |
| 默认卡 | 看到饥饿 → 烤面包 |
| 可替换行动 | 唱歌 / 分面团 / 办试吃会 / 写诗 |
| 可观察动作 | 揉面、举托盘、站上箱子唱歌、把失败面团藏起来 |
| hard_rules | 不会故意让镇民挨饿；即使不烤面包也会试图安慰 |

#### 毛栗 · 邮差

| 项 | 内容 |
|---|---|
| 可见功能 | 信息传播、快讯、送信 |
| 隐性需求 | 想知道秘密，但也想被信任 |
| 默认卡 | 收到消息 → 立刻传播 |
| 可替换行动 | 私下询问 / 写快讯 / 暂时保密 / 夸张转述 |
| 可观察动作 | 背邮包奔跑、停在门口偷听、写标题、把信压在帽子下 |
| hard_rules | 不会恶意造谣，但会因为好奇夸张事实 |

#### 铃兰 · 镇长助理

| 项 | 内容 |
|---|---|
| 可见功能 | 秩序、排班、规则解释 |
| 隐性需求 | 想证明自己不只是规则的复读机 |
| 默认卡 | 发现混乱 → 制定表格 |
| 可替换行动 | 排队 / 开会 / 放假 / 道歉 / 睡懒觉 |
| 可观察动作 | 拿夹板、贴公告、擦掉公告、深呼吸、把铃铛收起来 |
| hard_rules | 不会主动制造危险；即使放松也会担心后果 |

#### 洛洛 · 修理工

| 项 | 内容 |
|---|---|
| 可见功能 | 修设施、发明、处理环境问题 |
| 隐性需求 | 想做自己的发明，不只是修别人弄坏的东西 |
| 默认卡 | 机器坏了 → 慢慢修 |
| 可替换行动 | 快修 / 做替代工具 / 解释原理 / 暂停观察 |
| 可观察动作 | 拧螺丝、趴进机器、掏出奇怪零件、画发明草图 |
| hard_rules | 不会声称修好了没修好的东西；但可能做出奇怪替代品 |

## 8. 行为卡系统

### 8.1 卡片结构

```json
{
  "card_id": "aju_when_hungry_sing",
  "owner": "aju",
  "trigger": "看到有人饥饿",
  "action": "唱歌安慰",
  "tone_tag": "温柔地",
  "intent_hint": "想让大家觉得还能撑一会儿",
  "mechanical_effect": {
    "satiety": -6,
    "order": -2,
    "affection": +8,
    "weirdness": +6
  },
  "risk": "饱腹过低时会被认为不负责任",
  "visible_action": "阿橘站上面粉箱唱起了早餐歌。"
}
```

### 8.2 玩家可改范围

P0 只允许：

| 操作 | 例子 |
|---|---|
| 替换 action | 烤面包 → 唱歌安慰 |
| 加 tone_tag | 温柔地 / 慌张地 / 秘密地 / 隆重地 |
| 恢复默认卡 | 唱歌安慰 → 烤面包 |

不允许：

- 自由输入新 trigger；
- 自由输入长规则；
- 改 hard_rules；
- 直接改数值。

### 8.3 语气标签的作用

同一行动通过语气标签变成不同体验：

| 行动 | 标签 | 可能结果 |
|---|---|---|
| 唱歌 | 温柔地 | 亲密 +，秩序小降 |
| 唱歌 | 隆重地 | 荒诞 +，毛栗更可能传播 |
| 唱歌 | 秘密地 | 只影响附近 Agent，可能生成秘密记忆 |
| 唱歌 | 慌张地 | 秩序下降，铃兰更可能介入 |

## 9. 多 Agent 回合结构

每个时段按 8 步执行：

```text
1. 读取工单与当前小镇状态
2. 找出每个 Agent 的触发卡
3. Agent 根据人设/记忆/关系解释规则
4. Agent 输出结构化行动
5. 状态机结算指标与关系
6. 选择 2-4 个显著事件进入事件流
7. Agent 群聊互相回应
8. 压缩记忆，写入因果日志
```

### 9.1 Agent 输出 Schema

```json
{
  "agent_id": "aju",
  "chosen_action": "sing_to_hungry_neighbors",
  "target": "hungry_neighbors",
  "visible_action": "阿橘站上面粉箱唱起了早餐歌。",
  "speech": "如果面包还没醒，那我先让大家的胃听一首前奏！",
  "confidence": 0.64,
  "rule_alignment": "followed",
  "memory_to_add": "玩家让我在饥饿时唱歌而不是烤面包。"
}
```

### 9.2 rule_alignment

| 值 | 含义 |
|---|---|
| followed | 按规则执行 |
| hesitated | 犹豫执行 |
| adapted | 根据人设改写执行方式 |
| resisted | selfhood 高时轻微抵触 |

P0 中 resisted 只在夜晚或高 selfhood 时出现，避免首局失控。

## 10. 小镇指标与关系

### 10.1 全局指标

| 指标 | 范围 | 玩家理解 |
|---|---:|---|
| 饱腹 satiety | 0-100 | 大家有没有吃上东西 |
| 秩序 order | 0-100 | 小镇是否按规则运行 |
| 亲密 affection | 0-100 | 居民之间是否更愿意互相理解 |
| 荒诞 weirdness | 0-100 | 规则是否产生有趣偏差 |

删掉 v0.3 的 hunger/repair/rumor/happiness 多指标组合，P0 不需要这么细。

### 10.2 Agent 变量

| 变量 | 范围 | 说明 |
|---|---:|---|
| mood | 0-100 | 影响语气 |
| trust_player | -100~100 | 是否相信玩家改规则有善意 |
| selfhood | 0-100 | 是否会轻微越界 |
| fatigue | 0-100 | 避免同一 Agent 反复承担所有问题 |

### 10.3 关系矩阵

关系变化要有可读原因。

| 事件 | 关系变化 |
|---|---|
| 毛栗替阿橘解释“歌不是早餐” | 阿橘→毛栗 +5 |
| 铃兰公开批评阿橘不烤面包 | 阿橘→铃兰 -5，秩序 +3 |
| 洛洛修好面包炉 | 阿橘→洛洛 +8，饱腹 +10 |
| 毛栗夸张传播“早餐歌危机” | 铃兰→毛栗 -6，荒诞 +5 |

## 11. 涌现事件表

P0 不追求无限涌现，而是做 10-15 个强公式。

| ID | 条件 | 事件 | 效果 | LLM 表现 |
|---|---|---|---|---|
| E01 | 阿橘不烤面包 + 饱腹 < 40 | 早餐抱怨 | 秩序 -5，亲密 -3 | 镇民排队但没人拿到面包 |
| E02 | 阿橘唱歌 + 亲密 > 55 | 早餐歌扩散 | 亲密 +6，荒诞 +5 | 毛栗写快讯 |
| E03 | 洛洛快修 + fatigue 高 | 修理失败 | 饱腹 -5，荒诞 +4 | 面包炉吐出蓝烟 |
| E04 | 洛洛慢修 + 铃兰秩序高 | 排队修炉 | 秩序 +5，饱腹 -3 | 铃兰贴维修排班 |
| E05 | 毛栗传播 + 荒诞 > 50 | 谣言化 | 荒诞 +8，秩序 -4 | “早餐歌能当早餐”流传 |
| E06 | 铃兰放假 + 饱腹低 | 懒散危机 | 秩序 -8，亲密 +4 | 大家开心但更饿 |
| E07 | 铃兰开会 + 饱腹低 | 会开太久 | 秩序 +3，饱腹 -5 | 阿橘肚子叫被记录进会议纪要 |
| E08 | 阿橘唱歌 + 洛洛替代工具 | 吐司机留声机 | 饱腹 +4，荒诞 +10 | 洛洛做出会唱歌的吐司机 |
| E09 | 同一 Agent 连续被改 2 次 | 被改写记忆 | selfhood +10 | Agent 日记提到“不确定哪个才是我” |
| E10 | selfhood > 阈值 + hidden_need 相关 | 轻微越界 | 解锁灵魂片段 | Agent 做一件规则没写但符合自我的事 |

## 12. 自我值 selfhood

### 12.1 设计目的

表达主题：

> 规则能塑造人，但人不会完全等于规则。

### 12.2 增长来源

| 来源 | 增长 |
|---|---:|
| 同一 Agent 被连续改卡 | +8 |
| 行动与 hidden_need 相关 | +6 |
| 被其他 Agent 误解或支持 | +4 |
| 夜报中被记录为显著事件主角 | +5 |

### 12.3 首局可见的轻微越界

不要等长期模式才出现。首局如果阿橘 selfhood 足够高：

```text
夜晚，阿橘没有等新的行为卡。
他把烤焦的面包摆成一圈，在中间写了一句诗：
“如果面团会发酵，规则也许也会。”
```

这不是大剧情，只是让玩家意识到 Agent 开始记得并解释自己。

## 13. 鹳先生重新定位

鹳先生不是旁白主角，而是“规则审计员”。

| 场景 | 鹳先生负责 |
|---|---|
| 开局 | 解释工单和规则限制 |
| 时段后 | 给 1 句因果提示，不抢 Agent 话 |
| 夜报 | 整理因果链和指标变化 |
| 失败时 | 给调试建议 |

### 夜报结构

```text
今日工单：早餐危机
结果类型：温暖但饥饿

指标变化：
饱腹 45 → 38
秩序 65 → 61
亲密 50 → 64
荒诞 20 → 47

因果链：
你把阿橘的“烤面包”改成“唱歌安慰”
→ 阿橘在广场唱歌
→ 毛栗把它写成快讯
→ 铃兰认为歌不能登记为早餐
→ 洛洛开始研究会吐司的留声机

Agent 日记：
阿橘：我第一次觉得，面包也许不是我唯一能给大家的东西。
```

## 16. 结局 / 结果类型

首局不是传统胜负，而是结果画像。

| 结果 | 条件 | 体验 |
|---|---|---|
| 稳定小镇 | 饱腹和秩序较高，荒诞低 | 规则有效，但故事普通 |
| 温暖小镇 | 亲密高，饱腹/秩序中等 | 居民互相理解，代价是效率 |
| 混乱小镇 | 荒诞高，秩序低 | 很好笑，但问题没完全解决 |
| 规则醒来 | selfhood 事件触发 | Agent 出现规则外自我表达 |

每个结果都不是惩罚，而是玩家调试风格的回声。

## 18. 失败信号与修正

| 失败信号 | 原因 | 修正 |
|---|---|---|
| 玩家觉得只是换牌 | Agent 互相影响不足 | 增加关系变化、群聊引用、事件公式 |
| 玩家不知道为什么输/赢 | 因果链不清 | 夜报明确“你改了 X → 发生 Y” |
| Agent 像聊天机器人 | 可观察动作不足 | 每次输出必须包含 visible_action |
| LLM 编飞 | Prompt 太自由 | 限 allowed_actions + schema |
| 指标太多看不懂 | 模拟经营复杂度过高 | P0 只保留 4 指标 |
| 玩家不在乎居民 | hidden_need 不可见 | 夜报日记和 selfhood 小纸条提前暴露 |
| 鹳先生抢戏 | 旁白总结太强 | 群聊和 Agent 日记优先，鹳先生只做审计 |

## 20. 最终 30 秒 Demo 讲法

> 《编程小镇：规则醒来的那一天》是一个多 Agent 规则沙盒。玩家是鹳谷小镇的规则管理员，不能直接命令居民，只能改他们的 if-then 行为卡。比如把面包师阿橘的“看到饥饿就烤面包”改成“看到饥饿就唱歌”，AI 居民会根据人设、记忆和关系解释这条规则：有人被安慰，有人投诉，有人传播成谣言，修理工甚至可能做出会吐司的留声机。状态机负责事实和数值，LLM 负责居民如何理解、互相回应和记住这一天。最后夜报会回放玩家改动造成的因果链，让玩家看到一个小社会如何从规则里长出故事。

---

## 21. 一句话定稿

> **这不是一个教玩家写代码的游戏，而是一个让玩家看见“规则如何被活人误解、协商、反抗，并长成社会”的多 Agent 小镇沙盒。**
